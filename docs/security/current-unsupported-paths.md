# Current unsupported paths

Pipelock is an application-layer firewall. Its enforcement comes from sitting on the network path between the agent and the upstream. Traffic that bypasses the network path bypasses Pipelock. This document lists the egress paths the current binary does not intercept and the integrator action required for each.

The point of writing this down is honesty. Documenting the gaps gives the integrator a checklist. Soft-pedaling them produces a false sense of coverage.

## Raw socket egress

Processes that open a TCP or UDP socket directly without going through the configured HTTP proxy, HTTPS proxy, or SOCKS proxy will not be inspected. Examples include applications written against `net.Dial` in Go, `socket()` in C, `socket.socket()` in Python with no proxy library, anything using `nc` or `netcat`, and anything using `iperf`.

**Integrator action.** Apply egress firewalling at the OS or container level. Shipped options include `pipelock sandbox` for supported local subprocesses and the companion-proxy topology generated by `pipelock init sidecar` for Kubernetes deployments. Other deployments must enforce the same boundary with host firewall rules, container network policy, or equivalent controls. A workstation that runs Pipelock but allows arbitrary outbound TCP from any process still leaks to anything that ignores the proxy environment.

## Browsers without explicit proxy configuration

If a user opens Chrome, Firefox, Safari, or Edge on the workstation and the browser is not configured to use Pipelock as its HTTP and HTTPS proxy, all browser traffic bypasses Pipelock. The current binary does not transparently intercept system-wide traffic.

**Integrator action.** Use system-wide proxy configuration, group policy on Windows, PAC files, or an OS/container boundary that restricts browser egress to Pipelock. Do not rely on user-set proxy environment variables for browsers unless the deployment also prevents direct network egress.

## Processes that ignore `HTTPS_PROXY` and `NO_PROXY`

Some tooling does not honour the standard proxy environment variables. Examples include `git` without `http.proxy` configured, `curl` invoked with `--noproxy`, language SDKs with their own proxy logic, and container runtimes that do not propagate proxy settings to child processes.

**Integrator action.** Enumerate the tooling on the workstation, set explicit proxy configuration per tool, and audit periodically. The Pipelock home page documents which agent hosts are wired by the `pipelock claude install`, `pipelock cursor install`, and `pipelock jetbrains install` commands.

## MCP stdio servers without Pipelock wrapping

MCP servers spoken to over a child-process stdio pipe are not on the network. Pipelock's MCP scanning runs through the `pipelock mcp proxy --` wrapper, which executes the MCP server as a subprocess and inspects messages on the JSON-RPC boundary. An MCP server invoked directly by the agent host, without the wrapper, is not inspected.

**Integrator action.** Every MCP server invocation must run through `pipelock mcp proxy -- COMMAND ARGS`. The `pipelock claude install`, `pipelock cursor install`, and `pipelock jetbrains install` subcommands rewrite first-party agent host configuration to insert the wrapper. For agent hosts that are not first-party, the operator inserts the wrapper by editing the host configuration directly.

## UDP and direct DNS egress

Pipelock inspects HTTP and HTTPS. It does not inspect DNS lookups or arbitrary UDP. DNS is allowed to leave the host so that name resolution works at all.

**Integrator action.** If DNS-based exfiltration is in the threat model, use a DNS-layer firewall such as `dnsdist`, `unbound` with a blocklist, or AdGuard Home. Alternatively, run the agent inside a network namespace where DNS is constrained to a known resolver. UDP egress to non-DNS destinations should be blocked at the host firewall.

## Processes that mint their own CA trust

TLS interception requires the agent process to trust the Pipelock CA. A process that hard-codes its own CA bundle, for example some Go binaries with embedded roots or Python tooling with `REQUESTS_CA_BUNDLE` overriding the system store, will see Pipelock's intercepted certificate as untrusted. The process either fails closed, which is the desired outcome, or falls back to a non-inspected path if `--insecure` is set, which defeats the whole inspection model.

**Integrator action.** Review tooling for hard-coded CA bundles. Set `SSL_CERT_FILE`, `REQUESTS_CA_BUNDLE`, or the tool-specific equivalent to point at the combined CA bundle that includes the Pipelock root. Audit any flag that disables certificate verification.

## What is in scope

Pipelock's current enforcement covers HTTP, HTTPS via CONNECT, HTTPS with TLS interception when `pipelock tls init` has been run and the CA is trusted by the agent, WebSocket frames over the wrapped HTTPS path, MCP stdio when the agent host invocation routes through `pipelock mcp proxy --`, and MCP HTTP and SSE upstreams when the same wrapper is configured with `--upstream URL`. The supported-transport table in the project README is the authoritative list.
